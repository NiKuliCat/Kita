# AGENTS.md

本文档用于定义本项目的长期协作规范，供后续进入仓库的 AI agent 与开发者参考。

使用方式：

- 本文件只记录长期有效的项目约定。
- 临时任务要求不要写在这里，应保留在当前对话或任务说明中。
- 如果项目结构、流程或边界发生长期变化，应同步更新本文件。
- 新开窗口或新对话时，后续 agent 应先阅读本文件，再开始修改代码。

## 1. 项目概况

- 项目简介：这是一个基于 OpenGL 图形 API、ImGui 以及其他第三方库开发的桌面渲染引擎项目，后续计划逐步扩展为 3D 游戏引擎。
- 项目框架：项目分为“前端”和“后端”两层。
- `renderer/` 作为可执行项目，负责窗口、编辑器界面、调试视口与上层交互逻辑。
- `engine/` 作为静态库，负责渲染封装、资源加载、平台层、事件系统、场景组件等底层能力。
- 当前平台：目前以 Windows 平台开发与验证为主。
- 第三方库：当前主要依赖 `glm`、`imgui`、`assimp`、`spdlog`、`glad`、`glfw`。

## 2. 项目框架补充

- 项目采用“Renderer 可执行程序 + Engine 静态库”的结构，后续修改应尽量遵守这一层次边界。
- `renderer/` 中的代码应优先负责编辑器、视口展示、调试逻辑和引擎调用，不应反向承载大量底层实现。
- `engine/` 中的代码应优先承载可复用的底层能力，不应直接依赖编辑器层逻辑。
- 若后续新增模块，应优先考虑其属于 `renderer` 还是 `engine`，避免职责混杂。
- 如果项目目录结构、模块边界、核心职责发生长期变化，修改完成后应同步更新本文件。

## 6. 构建与验证要求

- 修改完成后，一般至少需要验证 `Debug|x64` 配置可以成功编译，除非我明确要求验证其他配置。
- 如果改动涉及构建脚本、依赖链接、运行库设置、资源拷贝或发布相关逻辑，应额外检查 `Release|x64` 是否仍可编译。
- 如果无法完成构建或验证，需要在结果中明确说明未验证项和原因。

## 7. 构建系统

- 当前构建系统以 Premake 为源头，`Build.lua`、`engine/engine_core_build.lua`、`renderer/renderer_build.lua` 是主要构建配置文件。
- `Setup.bat` 是生成 VS2026 工程的入口脚本。
- 默认应优先修改 Premake 配置，而不是把 `.vcxproj` 作为长期维护入口。
- 如果因为排查问题临时查看或调整生成后的工程文件，应尽量把最终规则回写到 Premake 脚本中。

常用构建约定：

- 工程文件生成命令：运行 `Setup.bat`
- 当前推荐解决方案：`Kita.slnx`
- 默认启动项目：`Renderer`
- `Engine` 为静态库，`Renderer` 为可执行程序

## 8. 依赖与链接补充

- `assimp` 当前采用本地二进制方式接入，相关文件放在 `engine/third-party/assimp/bin/Debug` 与 `engine/third-party/assimp/bin/Release`。
- 若依赖路径、命名规则、后处理拷贝逻辑发生变化，应同步检查 Premake 配置与本文件说明。
- 没有明确要求时，不要随意升级或替换第三方库版本。
